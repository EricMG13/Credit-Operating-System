# CAOS — Credit Agent OS

Credit analysis workspace: a six-concept analyst UI (Command Center,
Pipeline, Deep-Dive, Model Builder, Report Studio, Monitor) plus a global
**Ask (⌘K)** launcher, backed by a FastAPI service. Ships as a single
container — one process serves the JSON API under `/api` and the
statically-exported Next.js frontend at `/`. The supported deploy path is the
self-hosted **Docker stack** (`caos/deploy/`).

```
caos/
  server/      FastAPI app — serves /api and the built UI
    static/    Built frontend (generated by scripts/build_frontend.sh)
  frontend/    Next.js 15 app, output: "export"
  deploy/      Self-hosted Docker stack (Dockerfile, docker-compose, proxy)
  scripts/     build_frontend.sh
  tests/       server API tests + Playwright e2e
```

## Modules

The engine routing index ([`server/engine/registry.py`](server/engine/registry.py))
is the source of truth for what runs: **19 implemented analytical modules**
(CP-0, CP-1/1A/1B/1C, CP-2/2B/2C/2D/2E/2F, CP-3/3B/3C/3D, CP-4/4C, CP-6A/6E),
gated by the **CP-5 QA phase** (CP-5B lineage + the deterministic CP-5 severity
gate). Four further corpus modules are registered **spec-only**
(`implemented=False`, never executed) so the CP-X route plan reflects the full
methodology mesh honestly: **CP-SR**/**CP-MON** (L7 sector / monitoring) and
**CP-RENDER**/**CP-EXTRACT** (Infra). The `Modular OS/` corpus documents the
broader methodology; only the registry list above is wired in the engine.

## Architecture

| Concern        | Implementation                                                       |
|----------------|----------------------------------------------------------------------|
| Auth           | The app trusts forwarded identity headers (`X-Forwarded-User` / `-Email` / `-Preferred-Username`) set by an authenticating edge (**oauth2-proxy** in the Docker stack), layered with an in-app analyst profile login. The gate fails closed in production. |
| Database       | SQLite by default (ephemeral); set `DATABASE_URL` to any Postgres (`postgresql+asyncpg://…`) for durability — the Docker stack bundles one. Schema is managed by **Alembic** (migrations run on boot). |
| Document vault | Local dir by default; set `CAOS_STORAGE_DIR` to a durable mount — the Docker stack uses a volume at `/vault`. |
| LLM chat       | Anthropic Claude (`claude-opus-4-8`) via `ANTHROPIC_API_KEY` (env / secret). Without a key the chat endpoint returns canned demo replies. |
| Frontend       | Next.js static export served by FastAPI's StaticFiles — no Node process in deployment. |

## Deploy (self-hosted Docker)

The full launch runbook is [docs/LAUNCH_PHASE1.md](docs/LAUNCH_PHASE1.md); the
stack and ops reference live in [deploy/README.md](deploy/README.md). Quick start
on a Linux host with Docker + Compose v2:

```bash
cd caos/deploy
cp .env.example .env       # fill in domain, Postgres pw, the required secrets
                           # (SESSION_SECRET, EDGE_PROXY_SECRET, ANALYST_SIGNUP_CODE),
                           # Anthropic + EDGAR + Google OAuth — production fails
                           # closed at boot if the secrets are unset
docker compose up -d --build
```

This builds the app image (Next.js export baked in), runs Postgres, oauth2-proxy
(Google Workspace OIDC), and Caddy (TLS) — `Caddy → oauth2-proxy → app → db`.
The app self-migrates on boot, so a fresh Postgres needs no manual schema step.
Open `https://<CAOS_DOMAIN>`.

## Local development

Two options:

**Single process (what production runs):**

```bash
python3 -m venv .venv && .venv/bin/pip install -r server/requirements.txt
./scripts/build_frontend.sh
cd server && ../.venv/bin/python run.py     # http://localhost:8000
```

**Hot reload (UI dev):** run the API and `next dev` side by side — the dev
server proxies `/api` to :8000.

```bash
cd server && ../.venv/bin/python run.py &
cd frontend && npm install && npm run dev   # http://localhost:3000
```

## Tests

```bash
# Server API tests
.venv/bin/pip install pytest httpx
.venv/bin/python -m pytest tests/server -q

# Frontend unit tests
cd frontend && npm test

# E2E (needs the single-process server running on :8000)
cd frontend && npm run test:e2e
```
