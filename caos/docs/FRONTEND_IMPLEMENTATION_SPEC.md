# CAOS Frontend Implementation Spec (for Opus 4.8)

> **What this is.** The build order for raising every CAOS surface to the Impeccable *Excellent* band (≥36/40 critique) and wiring the AI leverage that makes this a credit analyst's primary instrument rather than one more dashboard. Execute **top-to-bottom by severity** (P0 → P3). Each item gives you the gap, the real files and named blocks (never line numbers — the tree moves under you), the Impeccable rule it serves, the exact instruction, and the one-line credit-market payoff. Where a design decision was mine to make, it is made — build the specified direction, not a menu.

## How to read this spec

- **Severity is the execution order.** P0 first (there are none open — see below), then P1 (the design-integrity and AI-leverage gaps that hold surfaces under 36), then P2 (polish, consistency, efficiency), then P3 (net-new material value — build only after the base is ≥36 everywhere).
- **Build the two shared primitives first.** The very first two P1 items define `LiveLaneState` (the four-state loading/empty/error/live shell) and `useRunPoll` (the run-poll/RE-RUN hook). Almost every other item references them by name. Build them before their adopters.
- **Reuse before invent.** Extend `components/shared/` primitives and the existing `--caos-*` tokens. There is no `components/ui` directory; new primitives are additions to `shared/`. Never add a dependency a few lines cover, never invent a token.
- **The mock↔engine seam is real.** Every "wire to engine" instruction names the exact `api.ts` call or `lib/engine` hook and its fallback. Several live lanes have **no `api.ts` client yet** — the instruction says so; add the typed client first, mirroring an existing one (`getPortfolio`/`getIssuer`). **No alert-stream/inbox/SSE endpoint exists** — nothing here is a live "stream"; "live" means a pull lane (digest, autonomy poll, `listRuns`), honestly labeled.
- **Report Studio is a fixed light "paper" theme.** Items touching it say so — never dark-theme `ReportDoc`; all new chrome goes in the dark `SubHeader`/rails. Every other surface is the fixed dark institutional-terminal system.
- **`Verified open at HEAD` line.** Each item carries a short note stating the current-tree evidence that its gap is real and unfixed. If you reach an item and find its gap already closed, skip it and say so — do not re-fix working code.

## Why these items (the two vectors)

The theming / anti-slop layer already passes the detector — reaching ≥36 is **not** a slop-removal job. The deficit lives in two places, and this spec concentrates there:

1. **The heuristic/UX layer** — system-status visibility, error prevention/recovery, recognition-over-recall, flexibility/power-paths, and help/documentation. The whole-app critique (26/40) sits at 2 on heuristics 5, 6, 8, 9, and 10.
2. **The AI-leverage layer** — Command Center and Monitor are seeded mock while a live autonomy DAG (`GET /api/autonomy/draft`) and daily-digest lane (`GET /api/digest/daily`) sit unwired. Converting those mock surfaces into live, AI-driven "what changed / why it matters" instruments is the single biggest lever for both goals: real live system-status is worth multiple heuristic points, and the drafted overnight finding is exactly the proprietary signal a Bloomberg terminal will not give the analyst.
