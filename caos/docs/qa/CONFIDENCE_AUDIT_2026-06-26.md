## [00] HIGH · confirmed_problem · ✓ verified
**Adjusted add-backs stripped from REPORTED EDGAR EBITDA — violates 'reported is canonical' invariant**
_Subsystem: deterministic credit-engine financial math (caos/server/engine)  ·  category: data-provenance  ·  investigator confidence: high_

**Root cause.** `reconcile_adjusted_ebitda` (adjusted.py:142-190) reconstructs EBITDA as `ebitda = nd/lev` from CP-1's `net_leverage_adj_ltm` and `net_debt_ltm`, then computes `ebitda_excl = ebitda*(1-pct)` (lines 160-161) where `pct` is an add-back load defined as a fraction of *adjusted* EBITDA (the regex/LLM both target "% of (adjusted) EBITDA"; LLM prompt at adjusted.py:111-113 says "FRACTION of adjusted EBITDA"). This is only valid when the reconstructed `ebitda` is the covenant-ADJUSTED figure. On the EDGAR path it is NOT: edgar_cp1.py:197 builds EBITDA = operating income + D&A + impairments (a reported GAAP proxy), edgar_cp1.py:232 sets leverage = net_debt/that-reported-EBITDA, and edgar_cp1.py:234 stuffs it into the same `net_leverage_adj_ltm` field (a misleading name — for EDGAR it holds a *reported* value, basis="reported_gaap_xbrl" at edgar_cp1.py:239). Root cause: `reconcile_adjusted_ebitda` is invoked unconditionally for every CP-1 in runner.py:139-148 with no `basis` guard (confirmed: grep finds zero `reported_gaap_xbrl`/basis checks in adjusted.py or the CP-1 wiring), even though the module's own docstring (adjusted.py:1-19) is written for an adjusted-basis input. The fixture/LLM path is correct because there `net_leverage_adj_ltm` genuinely is adjusted (fixtures.py:170-172: adj_ebitda 421, net_debt 2391, 2391/421=5.68x labelled "Adjusted net leverage"); the EDGAR path silently reuses the same arithmetic on a reported base.

**Impact.** When a US public filer (has ticker → EDGAR CP-1, basis=reported_gaap_xbrl) also has an uploaded credit agreement/offering memo disclosing an 'N% of adjusted EBITDA' add-back load, the engine publishes a fictitious 'quality-of-EBITDA' deleveraging gap to committee: a C-ADJ1 claim and a MINOR CP-1A-RECON QA finding asserting 'net leverage would be ~Xx vs reported' where Xx is strictly worse than the genuine ex-add-back (reported) leverage. The add-backs are double-counted: the engine treats the already-clean reported figure as if it were flattered and re-strips a haircut that was never there, overstating the credit's true leverage and the 'add-back risk' signal. This violates the 'reported is canonical' provenance invariant — a reconciliation that should be a no-op (or skipped) on reported-basis CP-1 instead manufactures a worse-than-reported number a credit analyst could rely on in an IC memo. Blast radius is bounded by co-occurrence (EDGAR issuer AND ingested agreement with a parseable add-back %); the seeded demo issuers have tickers but no add-back chunk in their seed text, so it does not trip today's fixtures, but it is a guardless real-data hazard, and the standard credit-agreement phrasing ('add-backs of N% of Consolidated Adjusted EBITDA') is exactly what the regex/LLM target.

## [01] HIGH · confirmed_problem · ✓ verified
**latest()/year() mis-order series that mix 2-digit and 4-digit year labels (FY2024 vs LTM_Q1_26)**
_Subsystem: deterministic credit-engine financial math (caos/server/engine)  ·  category: correctness  ·  investigator confidence: high_

**Root cause.** year() compares trailing year-digit runs of unequal width as raw integers, so a 2-digit period (LTM_Q1_26->26) sorts below a 4-digit one (FY2024->2024); latest() and the six other year()-ordered consumers inherit this, and the only producer that can emit a mixed-width series in one dict — the un-normalized LLM/live CP-1 (synth.py:444, schema synth.py:165) — is a reachable production path.

**Impact.** When a live (LLM) CP-1 run emits a mixed-width period series — e.g. revenue/adj_ebitda labeled {'FY2024': ..., 'FY2023': ..., 'LTM_Q1_26': ...} — every overlay that reads the 'latest' EBITDA silently takes the STALE prior fiscal year instead of the most recent LTM period. Concretely: (1) Recovery EV in CP-3B (capstructure._distressed_ev) = 5x the wrong (older, usually lower) EBITDA → every tranche's expected-recovery % is computed against an understated distressed EV. (2) Interest runway in CP-2E (liquidity._interest_runway_months) derives cash interest from the stale EBITDA → wrong months-of-liquidity-covers-interest. (3) Macro rate-shock sensitivity (macro.compute_rate_sensitivity) uses stale EBITDA as the base. (4) Peer ebitda_margin (peers._own_values) computes margin at max(year) period → wrong period, wrong percentile, potentially a spurious/missed bottom-quartile outlier CP-5 finding. (5) Worst: CP-1B EarningsDelta (the 'what changed' module) inverts — a real +10% revenue/+15% EBITDA quarter is reported as a -9%/-13% YoY DECLINE with false 'Revenue declined' / 'Adjusted EBITDA declined' monitoring signals and the periods printed backwards. An analyst building a defensible IC view would be reading a fabricated negative-momentum read off a fundamentally improving issuer — exactly the 'money is behind a wrong read' failure mode. It does NOT fire on any current provenance (EDGAR 4-digit-only, fixture 2-digit-only, reported single-key), so today's runs are clean; it is armed the moment a live LLM CP-1 mixes label widths in one series, which the schema actively permits.

**Fix.** Two-line root fix in periods.py: normalize year width inside year() before comparison — e.g. pad a 2-digit value to a 4-digit base (y = int(nums[-1]); return 2000 + y if y < 100 else y). That alone re-orders all consumers correctly (latest({'FY2024':100,'LTM_Q1_26':200})->200) and is provenance-agnostic. Caveat: a bare 2-digit '99' would map to 2099, but no real label uses pre-2000 years, and the alternative (preferring explicit LTM like _headline_period) only helps the LTM-prefixed case, not 'Q1_26'/'FY2024' mixes. Belt-and-suspenders: normalize CP-1 period labels once at the synth boundary (in _payload_from_data or a CP-1 post-validate step) to a canonical 4-digit form so every downstream — not just year() — sees consistent widths, and add a regression test asserting latest()/compute_deltas on a mixed-width dict pick the true latest period.

## [02] HIGH · confirmed_problem · ✓ verified
**Recovery waterfall and liquidity totals rely on a ±120-char nearest-amount heuristic that can attribute the wrong tranche's size**
_Subsystem: deterministic credit-engine financial math (caos/server/engine)  ·  category: correctness  ·  investigator confidence: high_

**Root cause.** The heuristic's safety assumption is empirically false in this codebase. `amount_musd` (caos/server/engine/textscan.py:27-35) binds a tranche/source keyword to the FIRST `$` amount inside a window that starts 120 chars BEFORE the keyword (`text[max(0, m.start()-120): m.end()+120]`, line 30) — so it grabs whichever amount appears earliest in reading order in that window, which is systematically the *preceding* tranche's figure, not "nearest." The code comment (textscan.py:24-26) argues this is safe because "real agreement packs chunk tranches separately." But the actual chunker, `chunk_text` (caos/server/ingest.py:177-186), is a blind fixed-size character splitter: `CHUNK_CHARS = 2400`, step 2160 (line 32-33), with NO sentence/paragraph/table boundary awareness. Both production ingestion paths call it unchanged (routes/ingestion.py:73 for uploads, routes/edgar.py:190 for EDGAR exhibits). A typical cap-structure paragraph or "Description of Indebtedness" section (200-1200 chars) — the exact place all tranches are named together — therefore lands entirely in ONE chunk. BM25 (retrieval.py:74-94) ranks that multi-tranche chunk highest because it contains every query term. scan_tranches (capstructure.py:44-50) and scan_liquidity (liquidity.py:30-35) then call amount_musd once per keyword against that single shared chunk, so multiple keywords pull from the same pool of `$` figures and get the wrong one. The dedupe in `scan` (textscan.py:46-52) only prevents a tranche from matching twice; it does nothing to isolate amounts within a shared chunk.

**Impact.** CP-3B RecoveryInstrumentPreference and CP-2E LiquidityMaturityAnalysis present financially wrong numbers as High-confidence, Directly-Sourced facts whenever a source document names two or more tranches/liquidity sources within ~2400 chars of each other — which is the norm for real cap-structure tables and 'Description of Indebtedness' sections that the BM25 query is specifically designed to retrieve. Concretely: per-tranche sizes, total_debt_musd (overstated ~48% in the repro), pct_of_structure, the absolute-priority recovery_pct per tranche, the instrument preference ranking, and disclosed_liquidity_musd / months_liquidity_covers_interest can all be wrong, with NO limitation flag and confidence='High'. A buy-side analyst reading the recovery waterfall or liquidity bridge for an investment-committee view would be relying on numbers that look sourced and certain but are misattributed. This is squarely the 'money is behind a wrong read' / data-provenance-integrity failure the project treats as paramount, and the modules feed downstream CP-6A debate.

## [03] HIGH · confirmed_problem · ✓ verified
**CP-6A IC debate verdict systematically omits CP-2B (downside fragility) and CP-3 (peer fundamentals) — they share CP-6A's layer, so upstream.get() always returns None**
_Subsystem: CP-X DAG orchestrator (engine/registry.py, planner.py, runner.py, gate.py, lineage.py, readiness.py, council.py, debate.py, synth.py, periods.py)  ·  category: correctness  ·  investigator confidence: high_

**Root cause.** CP-6A reads up.get('CP-2B') and up.get('CP-3') in _ic_signals (debate.py:143, 153), but its registry depends_on is only ('CP-1','CP-2','CP-4C') (registry.py:131) — CP-2B and CP-3 are NOT declared deps. The runner groups modules into dependency LAYERS from registry.depends_on (runner.py:_dependency_layers, 86-112). Because CP-6A's transitive depth (via CP-2/CP-4C) equals CP-2B's (via CP-2) and CP-3's (via CP-1C) — all three at depth 3 — they all land in dependency layer 2 (verified empirically). Within a layer, the runner synthesizes the non-_SESSION_SYNTH modules CONCURRENTLY via asyncio.gather over the SAME upstream dict (runner.py:318-324); upstream[module_id]=payload is assigned only AFTER the gather, in the sequential _persist_synth_result loop (runner.py:298, called from runner.py:325-326). CP-2B, CP-3 and CP-6A are all non-session modules, so they run in the same gather and CP-6A reads upstream before CP-2B/CP-3 are persisted into it — up.get('CP-2B')/up.get('CP-3') return None on EVERY run. This is invariant (independent of CP-0 sources or issuer data), because _dependency_layers consults only registry.depends_on, and the relative depths are fixed. The true underlying cause is a stale-comment / stale-dependency bug: the registry comment at registry.py:127-129 says CP-2B/CP-3 are 'spec-only ... which would force a Blocked verdict' and were deliberately left out of depends_on to avoid blocking CP-6A. But CP-2B and CP-3 were SUBSEQUENTLY implemented (implemented=True, commit 3a79978) and are now routed — the depends_on was never updated to match, so the 'opportunistic read' the author intended as a graceful-degrade became a guaranteed-miss.

**Impact.** The two most consequential outputs of the credit pipeline — the IC Chair credit verdict (CP-6A) and the CIO position-sizing posture (CP-6E, which consumes CP-6A's lean) — systematically ignore downside fragility (CP-2B) and peer fundamentals (CP-3) on every run. The dropped points are heavily weighted bear signals: HIGH downside fragility carries weight 3 (the single heaviest bear point in the module) and UNDERWEIGHT peer standing weight 2. With realistic inputs this flips the IC lean from CAUTIOUS to CONSTRUCTIVE (net +3 vs −2) and the sizing posture from 'Pass / minimal — risk-reward does not support a position' to 'Add — constructive entry, size toward target'. For a leveraged-credit analyst, this is the worst class of error in this platform: a fragile, peer-lagging issuer can be presented to the investment committee as a constructive add.
