# E1 — Stress/Scale Closure Report

2026-07-12. First real run of the stress harness (`caos/tests/stress/`,
built but never executed before this session per the plan's own note) plus
the missing multi-worker deploy config it was meant to validate.

## What shipped

- `run.py`: `WEB_CONCURRENCY` env var launches N uvicorn worker processes.
  Guarded (`validate_workers`, tested in `test_run_launcher.py`) to refuse
  `WEB_CONCURRENCY>1` against a non-Postgres `DATABASE_URL` — SQLite has no
  cross-worker claim coordination (`InProcessExecutor`'s own docstring is
  explicit about this). Defaults to 1 — today's shipped single-process
  posture is unchanged unless the operator opts in.
- `docker-compose.yml` / `.env.example`: `WEB_CONCURRENCY` passthrough +
  sizing guidance against Postgres's default `max_connections=100` (each
  worker opens its own 5+10-connection pool).

## What was already there (verified, not built this session)

Multi-worker deploy turned out to need *only* the launch flag — the hard
part was already done:

- `run_executor.get_executor()` already picks `QueueWorker` (Postgres,
  `FOR UPDATE SKIP LOCKED` claim) vs `InProcessExecutor` (SQLite) by DB
  dialect at runtime — genuinely safe for N>1 workers on Postgres.
- `database.py`'s `init_db()` already takes a Postgres session-level
  advisory lock around `alembic upgrade head`, explicitly documented as
  built for "concurrent boots" — i.e., this exact multi-worker case.

## What's still single-process-only (found this session, not fixed)

`research_executor.py` and `research_report_executor.py` (Deep Research
jobs, research-report generation) are still in-process-only even on
Postgres — each worker runs its own asyncio task with no cross-worker
claim/reclaim. Under normal multi-worker operation this is **not** a
double-execution risk (a request only ever reaches the one worker that
received it), but a worker that dies mid-job leaves that job "running"
forever until the next full restart's boot sweep — narrower than the
CRIT this loop closes, tracked separately (PR #150's lease-gated boot
sweep addresses the redeploy-timing half of this; full claim/reclaim
parity with `QueueWorker` is future work, not blocking E1).

## Stress run — real, against a throwaway Postgres 18 + 2 uvicorn workers

Setup: `pgvector/pgvector:pg18` in Docker, `WEB_CONCURRENCY=2`, seeded 30
issuers + 3 completed runs (`seed_stress.py`), both worker processes booted
clean — the alembic advisory lock did exactly what its docstring promises:
worker 1 ran the full migration chain, worker 2's log shows it skipped
straight to "already at head" with no DDL race.

**Load (`locustfile.py`, headless, 15 users, ramp 3/s, 60s, no LLM):**

| Metric | Result |
|---|---|
| Total requests | 2,584 |
| Failures | **0 (0.00%)** |
| Aggregate p50 / p95 / p99 | 27ms / 89ms / 130ms |
| `POST /runs/{id}/report` p95 | 90ms |
| `GET /runs/{id}/modules/{m}` p95 | 100ms (heaviest endpoint, as expected) |

**Verdict: PASS.** Zero failures at 15 concurrent users across the
expensive endpoints (report render, module detail, NL query) the harness
targets. No sign of pool exhaustion, no 5xx, no request queuing beyond the
per-endpoint compute cost itself.

**Fault injection (Anthropic timeout/backoff — STRESS_TEST_PLAN §7 HIGH
items):** not re-run this session — already verified fixed and documented
(`llm.py:65` sets an explicit `timeout=settings.caos_llm_timeout_s` on the
Anthropic client; `caos-llm-fault-isolation` invariant — every LLM lane
fault-isolated, 120s timeout, no lane can hang a run forever). Re-confirm
at the next full E1/G3 pass if `llm.py`'s client construction changes.
